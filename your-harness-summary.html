🤖 your-harness

v0.1.0 CLI-first TypeScript Node.js

your-harness es un arnés de desarrollo potenciado por inteligencia artificial, diseñado como herramienta de línea de comandos centralizada y extensible. Orquesta agentes, habilidades (skills), plugins, servidores MCP, workflows y especificaciones SDD/OpenSpec, con soporte para múltiples proveedores de IA.

🧱 Arquitectura General

El sistema sigue un diseño modular en capas, con fuerte separación de responsabilidades:

  1. CLI – Punto de entrada único (yh), construido con Commander y Chalk.
  2. Core – Núcleo funcional: config, logger, eventos, contexto, conectores, managers de IA y MCP, builder de prompts.
  3. Proveedores IA – Implementaciones concretas para Copilot, Claude, OpenAI, modelos locales (Ollama) y endpoints custom.
  4. Plugins & Skills – Sistema de plugins cargables dinámicamente y skills con templates e inyección de system prompts.
  5. Agentes – Ejecución autónoma con loop de razonamiento, uso de herramientas y límites de iteración.
  6. MCP – Cliente y servidor JSON-RPC 2.0 para integrar herramientas externas.
  7. Workflows – Motor de workflows con DAGs, pasos paralelos, condiciones y reintentos.
  8. SDD / OpenSpec – Parser multi-formato, validador y generador de código a partir de especificaciones.

⚡ Características principales

  • CLI contextual: Modos de trabajo (frontend, backend, devops, testing).
  • Multi-proveedor IA: Copilot, Claude, OpenAI, local (Ollama), custom. Selección por defecto, fallback automático y testeo interactivo.
  • Plugins dinámicos: Instalación desde ruta local o paquete npm, ciclo de vida con hooks, comandos exportables.
  • Skills componibles: System prompts inyectables, templates con variables, dependencias entre skills.
  • Agentes autónomos: Developer y Reviewer incluidos; loop de razonamiento con tool calling, límite de iteraciones, eventos de progreso.
  • MCP nativo: Cliente para conectar a servidores externos; servidor integrable que expone herramientas del harness.
  • Workflows complejos: Motor basado en DAG, pasos secuenciales/paralelos, condiciones, reintentos, timeouts, pipelines predefinidos (code‑review).
  • SDD/OpenSpec: Parseo de Markdown (con frontmatter YAML), YAML puro, JSON; validación estructural y generación de artefactos (tipos TypeScript, OpenAPI, scaffold, tests, documentación).
  • Arquitectura preparada para futuro: Sistema de memoria externo (vía MCP) previsto, sin acoplamiento.

✅ Puntos fuertes

  • ✔ Diseño modular y extensible – cada pieza puede reemplazarse o ampliarse sin romper el resto.
  • ✔ Enfoque funcional – código predecible, sin efectos secundarios ocultos, tipado estricto.
  • ✔ CLI completo – más de 25 comandos documentados y listos para usar.
  • ✔ Abstracción de IA – cambio de proveedor sin modificar agentes o workflows.
  • ✔ Preparado para MCP – tu sistema de memoria encaja como un servidor MCP externo que se auto‑registra.
  • ✔ Código base limpio y homogéneo – patrones consistentes en todos los módulos.

⚠️ Áreas de mejora (y limitaciones actuales)

  • ⚠ Las herramientas de los agentes (read_file, write_file, etc.) retornan respuestas simuladas; falta implementación real con fs y child_process.
  • ⚠ El transporte MCP está simulado – se necesita integrar stdio/subprocess y HTTP SSE reales.
  • ⚠ La escritura de configuración (YAML) es un placeholder; actualmente solo se lee y fusiona.
  • ⚠ No hay tests unitarios ni de integración implementados.
  • ⚠ La evaluación de condiciones en workflows es muy básica (sin analizador de expresiones seguro).
  • ⚠ El sistema de memoria (tu proyecto con Postgres+pgvector) está diseñado pero no integrado.
  • ⚠ No existe UI gráfica ni modo interactivo fuera de la terminal.

🗺️ Consideraciones a corto, mediano y largo plazo

🔹 Corto plazo (próximas semanas)

  • Implementar ejecución real de herramientas de agente (filesystem, comandos).
  • Integrar transporte MCP real (usando el SDK de MCP o implementación manual con child_process y fetch).
  • Escribir tests unitarios para los módulos core (config, eventos, registry, managers).
  • Completar la persistencia de configuración (lectura/escritura YAML real).
  • Añadir un par de skills/workflows adicionales como ejemplos concretos.

🔸 Mediano plazo (1‑3 meses)

  • Conectar tu sistema de memoria externo mediante MCP (memory-cli setup --harness).
  • Desarrollar un mecanismo sencillo de descubrimiento/instalación de plugins (por ej. desde npm o rutas locales).
  • Ampliar los triggers de workflows: watchers de archivos, git hooks, cron.
  • Crear una interfaz de usuario básica (TUI con Ink.js o Electron) para visualizar sesiones y workflows.
  • Agregar métricas de uso y telemetría opcional para mejora continua.

🔹 Largo plazo (6+ meses)

  • Soporte multi‑usuario y colaboración en equipo.
  • Sincronización de configuración y memoria en la nube.
  • Integración con IDEs (extensión de VS Code).
  • Aprendizaje continuo: el sistema mejora sus sugerencias basándose en sesiones anteriores.
  • Ciclo SDD completo: sincronización bidireccional entre especificaciones y código real.
  • Marketplace de plugins, skills y agentes comunitarios.

🧭 Próximos pasos sugeridos

Con la base sólida terminada, te recomiendo avanzar en este orden:

  1. 1. Implementar herramientas reales de agente – Dar vida a read_file, write_file, run_command usando fs y child_process. Esto hará que los agentes sean realmente funcionales.
  2. 2. Activar transporte MCP real – Adaptar el cliente MCP para usar stdio (spawn de proceso) y/o HTTP SSE. Así podrás conectar tu futuro memory-cli sin simulaciones.
  3. 3. Crear una suite de tests – Empezar con vitest probando config, eventos, registro de AI y managers. Esto blindará el núcleo.
  4. 4. Integrar tu sistema de memoria – Una vez que el cliente MCP funcione, conectar tu memory-cli como servidor externo y aprovechar sus tools desde los agentes.
  5. 5. Refinar la experiencia de CLI – Agregar prompts interactivos (inquirer), autocompletado y mejores mensajes de error.
